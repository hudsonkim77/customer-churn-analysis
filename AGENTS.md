# my-wiki-02

통신사 CS(고객센터) 데이터를 분석하고 인사이트를 축적하는 개인 분석 위키. 이 문서는 이 위키에서 작업할 때 지켜야 할 규칙을 정의한다.

## 폴더 구조

```
raw/            원본 데이터. 절대 수정하지 않는다.
00_inbox/       미분류 메모, 자유 형식 자료 요약, 검토 요청 메모
01_questions/   탐구할 질문
02_data/        raw 데이터의 스키마 노트
03_analysis/    질문에 답하기 위한 분석 결과
04_insights/    분석에서 얻은 인사이트
05_log/         날짜별 작업 로그
INDEX.md        전체 노트로 가는 허브 노트
```

각 폴더의 목적은 폴더 내 README.md에도 1~2줄로 정리되어 있다.

## 데이터 카탈로그

`raw/`에는 아래 5종의 CS 데이터가 있다. 분석 기간은 2024년 1월~12월.

| 파일 | 내용 | 건수 |
|---|---|---|
| data_customers.csv | 고객 마스터 | 500명 |
| data_voc.csv | VOC | 1,307건 |
| data_consultations.csv | 상담 로그 | 1,320건 |
| data_satisfaction.csv | 만족도 | 1,320건 |
| data_usage_history.csv | 이용 이력 | 6,000건 |

**연결 키**: `customer_id` 기준으로 대부분의 테이블이 연결된다. 단 `data_satisfaction`은 `consult_id`로 `data_consultations`와 연결한다.

## 원본 자료 처리 규칙 (raw/ 동기화)

- `raw/`의 파일은 절대 수정하지 않는다. 원본 그대로 보존한다.
- `raw/`에 새 파일이 추가되면, 먼저 `raw/` 파일 목록과 `02_data/`, `00_inbox/`의 노트 목록을 **파일명 기준으로만** 대조한다. 내용은 읽지 않는다.
- 이미 대응하는 노트가 있는 파일은 다시 읽지 않는다 (토큰 낭비 방지).
- 대응 노트가 없는 새 파일만 실제로 읽어서 분류한다.
  - 표 형태 데이터 → `02_data/`에 스키마 노트 생성
  - 표가 아닌 자유 형식 자료(메모, 텍스트 등) → `00_inbox/`에 요약 노트 생성
  - 내용이 불분명해서 분류 판단이 어려우면 **추측해서 노트를 만들지 않는다**. `00_inbox/`에 검토 요청 메모를 남긴다.
- 생성한 노트에는 반드시 `source: [[raw/원본파일명]]`을 남긴다.
- 공통 컬럼(예: `customer_id`)이 있는 데이터끼리는 위키링크로 서로 연결한다.
- 이 과정은 `/wiki-sync` 명령으로 실행한다.

## 노트 작성 규칙

- 모든 노트 상단에는 YAML 프론트매터로 날짜·분류·관련 데이터 등 메타데이터를 포함한다.
- 주제 태그(`tags`)를 프론트매터에 추가한다. (예: `이탈분석`, `채널효율`)
- 인사이트 노트는 **근거 → 해석 → 시사점** 순서로 작성한다.
- 노트 간 연결은 Obsidian 위키링크 `[[파일명]]` 형식을 사용한다.
- 추측과 사실을 명확히 구분한다. ("~로 보임" vs "~임")
- 새 노트가 생성될 때마다 `INDEX.md`의 해당 섹션(## 데이터 / ## 검토 필요 / ## 질문 / ## 인사이트)에도 링크를 추가한다.

각 노트 유형의 프론트매터 항목과 본문 구성은 `01_questions/_template.md`, `04_insights/_template.md`를 따른다.

## 작업 로그 규칙

세션이 끝나면 `05_log/오늘날짜.md`에 이번 세션에서 처리한 raw 파일과 생성한 노트 목록을 요약해서 기록하고, `INDEX.md`의 `## 최근 로그` 섹션에도 링크를 추가한다.

## 분석 기준

- 재문의율 20% 이상이면 개선 필요로 판단한다.
- 표본 30건 미만 세그먼트는 참고용으로만 사용하고, 결론의 근거로 쓰지 않는다.
